# Hospital ERP - Cross-Module API Integration Map

### 🌐 1. Module 1: ADM-MC (القبول والهوية الرقمية) - المركزية
*هذا الموديول هو "النواة"، لا يأخذ بيانات من أحد (إلا من المستخدم المباشر)، بل يعطي البيانات لجميع الأقسام.*

*   **يعطي (Gives to) Module 4 (الصيدلية):** 
    *   الـ API: `GET Endpoint 3: Risk Profile Access`
    *   البيانات: يرسل `allergies`, `chronic_diseases`, `blood_type` بناءً على الـ `digital_id_token`.
*   **يعطي (Gives to) Module 5 (العمليات):**
    *   الـ API: `GET Endpoint 3: Risk Profile Access` (نفسه الصيدلية للتأكد من سلامة التخدير).
    *   الـ API: `GET Endpoint 1: Token Retrieval` (للتحقق من هوية المريض قبل العملية).
*   **يعطي (Gives to) Module 2 (المالية):**
    *   الـ API: `POST Endpoint 4: Admission Syncing` (لفتح ملف مالي للمريض).
    *   الـ API: `POST Endpoint 7: Medical Coding` (إرسال أكواد ICD-10 لإصدار الفاتورة النهائية).
*   **يعطي (Gives to) Module 3 (الأسرة):**
    *   الـ API: `POST Endpoint 4: Admission Syncing` (لتأكيد قبول المريض وتخصيص سرير).
*   **يأخذ (Takes from) Module 6 (الطوارئ):**
    *   الـ API: `POST Endpoint 6: Emergency Temporary Identity` (يستقبل بيانات المريض المجهول من الطوارئ لإنشاء `temp_id` له).

---

### 💊 2. Module 4: PHM-LOG (الصيدلية)
*تعتمد على المركزية للتحقق، وترسل التكلفة للمالية، وطلب التزويد للمخازن.*

*   **يأخذ (Takes from) Module 1 (القبول):**
    *   الـ API: `GET Endpoint 3: Risk Profile Access` (للتحقق من ملف المخاطر قبل الصرف).
*   **يعطي (Gives to) Module 2 (المالية):**
    *   الـ API: `POST Endpoint 2: Send Dispensing Cost to Finance` (يرسل `digital_id_token`, `amount`, `medical_service_id` كفاتورة أدوية).
*   **يعطي (Gives to) Module 7 (المخازن):**
    *   الـ API: `POST Endpoint 3: Request Inventory Restock` (يرسل طلب تزويد أدوية عند وصول المخزون للحد الأدنى).
*   **يأخذ (Takes from) Module 7 (المخازن):**
    *   يستقبل تحديث الكميات (دخول أدوية جديدة للمخزون) لتحديث جدول `Medications`.

---

### 🚑 3. Module 6: ER-FLOW (الطوارئ)
*يتعامل مع الحالات الحرجة، يحتاج أسرة فوراً، ويسجل المرضى المجهولين.*

*   **يأخذ (Takes from) Module 1 (القبول):**
    *   الـ API: `POST Endpoint 6: Emergency Temporary Identity` (يحصل على `temp_id` للمرضى المجهولين).
    *   الـ API: `GET Endpoint 1: Token Retrieval` (للحصول على `digital_id_token` للمرضى المعروفين).
*   **يأخذ (Takes from) Module 3 (الأسرة):**
    *   يحتاج الاستعلام عن توفر أسرة الطوارئ الحرجة. *(ملاحظة للـ Integration Team: فريق الطوارئ حدد الـ `Endpoint 4 & 5` في الـ API الخاص بهم، هذا يعني أنهم يطلبون من فريق الأسرة توفير API مشابه للربط).*
*   **يعطي (Gives to) Module 1 (القبول):**
    *   يرسل بيانات المريض الأولية لتسجيله في النواة إذا كان جديداً.

---

### 🛏️ 4. Module 3: IPD-BED (الأسرة والإقامة)
*يدير الغرف ويرتبط ارتباطاً وثيقاً بالقبول، المالية، والعمليات.*

*   **يأخذ (Takes from) Module 1 (القبول):**
    *   الـ API: `POST Endpoint 4: Admission Syncing` (يستقبل إشعار بقبول مريض لتخصيص سرير).
*   **يعطي (Gives to) Module 5 (العمليات):**
    *   يوفر حالة السرير (متاح/مشغول) للتأكد من توفر سرير تعافي للمريض بعد العملية.
*   **يعطي (Gives to) Module 2 (المالية):**
    *   يرسل إشعار خروج متوقع قبل 24 ساعة (لإعداد الفاتورة).
    *   يرسل تكلفة الإقامة اليومية.

---

### 🏥 5. Module 5: SURG-OPT (العمليات)
*يحتاج تنسيق بين القبول (للمريض)، المخازن (للأدوات)، والأسرة (للتعافي).*

*   **يأخذ (Takes from) Module 1 (القبول):**
    *   الـ API: `GET Endpoint 3: Risk Profile Access` (للتأكد من سلامة التخدير والحساسية).
*   **يأخذ (Takes from) Module 3 (الأسرة):**
    *   يستعلم عن توفر سرير تعافي قبل جدولة العملية.
*   **يأخذ (Takes from) Module 7 (المخازن):**
    *   يستعلم عن توفر الأدوات والمستلزمات الجراحية قبل العملية.
*   **يعطي (Gives to) Module 2 (المالية):**
    *   يرسل تكلفة العملية الجراحية (كخدمة طبية `MedicalService`).

---

### 💰 6. Module 2: FIN-INS (المالية والتأمين) - محور التكلفة
*يستقبل التكاليف من الجميع، ولا يعطي بياناتاً سريرية لأحد، بل يعطي حالة الدفع والتأمين.*

*   **يأخذ (Takes from) Module 1 (القبول):** فتح ملف مالي، وأكواد الترميز (ICD-10).
*   **يأخذ (Takes from) Module 4 (الصيدلية):** فواتير الأدوية.
*   **يأخذ (Takes from) Module 3 (الأسرة):** فواتير الإقامة وإشعارات الخروج.
*   **يأخذ (Takes from) Module 5 (العمليات):** فواتير العمليات.
*   **يأخذ (Takes from) Module 7 (المخازن):** تقارير استهلاك المواد.
*   **يعطي (Gives to) جميع الأقسام:**
    *   الـ API: يوفر حالة التغطية التأمينية `payment_status` (مغطى/غير مغطى) لمنع تقديم الخدمات غير المغطاة قبل البدء بها.

---

### 📦 7. Module 7: INV-SUP (المخازن)
*المورد للصيدلية والعمليات، ويقوم بإدارة انتهاء الصلاحية.*

*   **يأخذ (Takes from) Module 4 (الصيدلية):**
    *   الـ API: `POST Endpoint 3: Request Inventory Restock` (طلبات تزويد الأدوية).
*   **يأخذ (Takes from) Module 5 (العمليات):**
    *   يستقبل طلبات تزويد المستلزمات الجراحية.
*   **يعطي (Gives to) Module 4 & 5 (الصيدلية والعمليات):**
    *   يؤكد توفر المواد ويخصمها من المخزون عند التسليم.
*   **يعطي (Gives to) Module 2 (المالية):**
    *   يرسل تقارير دورية باستهلاك المواد وتكاليفها.

---

### 🚫 ملاحظات الـ Integration Team (Modules بدون تكامل مباشر)
بناءً على تحليل الـ APIs والـ Client Requirements، هناك موديولات لا تتحدث مع بعضها مباشرة، وتمر ببعضها (Pass-through):
1.  **الصيدلية (4) والطوارئ (6):** لا يوجد API مباشر بينهما. الطوارئ تعالج المريض وتكتب وصفة، الصيدلية تتعامل مع "الوصفة" وتطلب الهوية من "القبول (1)".
2.  **الصيدلية (4) والعمليات (5):** لا توجد حاجة لربط مباشر. كلاهما يأخذ البيانات من القبول (1)، وكلاهما يرسل الفواتير للمالية (2).
3.  **الطوارئ (6) والمالية (2):** الطوارئ لا تُصدر فواتير مباشرة، بل تحول المريض إما للقبول (فيقوم القبول بفتح ملف مالي) أو للخروج.
